# CRITIQUE PRODUIT — AllocCheck v1.0.0
**Date** : 2026-04-02
**Cible** : Allocataires CAF (13,5M de foyers) qui veulent verifier leurs droits et contester les ecarts
**Methode** : Revue par 6 conseillers produit specialises (UX, Copy, Features, Onboarding, Monetisation, Coherence)
**Resume** : 18 observations (7 a revoir | 6 discutables | 5 points forts)

---

## Synthese executive

AllocCheck a un positionnement produit solide : le moteur de calcul local est serieux (sources legales citees, baremes avril 2026), le formulaire multi-etapes est bien pense avec un excellent systeme de checklist pour les revenus (montants fixes pre-remplis, filtrage contextuel). La proposition de valeur "simulation gratuite, contestation payante" est pertinente.

Les points a retravailler en priorite : (1) la divergence des baremes entre le moteur local Dart et l'Edge Function TypeScript, qui va generer des resultats differents selon la source, (2) le melange tutoiement/vouvoiement qui casse la coherence du ton, (3) les dependances declarees mais non utilisees (go_router, Riverpod) qui indiquent un scaffold inacheve avec des choix architecturaux non appliques.

---

## Points forts — choix pertinents confirmes

### 1. Moteur de calcul local avec sources legales
**Domaine** : Features
Le calcul 100% local dans `calcul_local_service.dart` est un choix pertinent : zero latence, fonctionne offline, et chaque bareme est documente avec son decret source. La constante `sourcesLegales` associe chaque aide a sa reference juridique exacte, ce qui est essentiel pour la credibilite aupres d'une cible qui va potentiellement contester la CAF.

### 2. Systeme de checklist pour les autres revenus
**Domaine** : UX
L'enum `TypeAutreRevenu` avec `montantFixe` et `saisieRequise` est une excellente decision. Les aides a montant fixe national (ASS, bourses CROUS) sont cochables sans saisie, avec le montant affiche comme badge de confirmation. Cela reduit massivement la friction et les erreurs de saisie.

### 3. Filtrage contextuel intelligent
**Domaine** : UX / Features
La methode `_filteredAutresRevenus()` masque les revenus non pertinents selon le contexte deja saisi (ex: pas d'ASS si emploi, pas de pension invalidite si aucun historique d'emploi). L'etape 4 masque l'AAH si le handicap est coche (calculee auto). Ce sont des choix produit matures.

### 4. Deconjugalisation AAH correctement implementee
**Domaine** : Features
Le calcul AAH prend en compte la deconjugalisation effective depuis octobre 2023 — seules les ressources du demandeur sont considerees. C'est un detail reglementaire important que beaucoup d'outils negligent.

### 5. Proposition de valeur freemium bien positionnee
**Domaine** : Monetisation
La simulation est gratuite (acquisition), le paiement intervient apres la decouverte de valeur (ecart detecte). Le paywall est au bon moment du parcours : l'utilisateur a deja vu son ecart potentiel avant qu'on lui propose de payer pour contester.

## A revoir — recommandations prioritaires

### 1. Divergence de baremes entre moteur local et Edge Function
**Domaine** : Features / Coherence
**Choix actuel** : Deux moteurs de calcul coexistent — `calcul_local_service.dart` (baremes avril 2026, RSA base 651.69 EUR) et `calculate-rights/index.ts` (baremes differents, RSA base 635.71 EUR). Les montants divergent sur toutes les aides (APL, AF, Prime activite, AAH).
**Probleme** : Un utilisateur qui simule sur l'app mobile et sur le web obtiendra des resultats differents. Cela detruit la confiance, surtout pour un outil dont la promesse est la precision.
**Proposition** : Faire du moteur local Dart la source de verite unique pour le MVP. L'Edge Function devrait etre un proxy qui appelle le meme jeu de baremes. A terme, stocker les baremes dans un fichier JSON versionne partage entre les deux codebases.

### 2. go_router et Riverpod declares mais non utilises
**Domaine** : Coherence / UX
**Choix actuel** : `go_router` (v14.8.1) et `flutter_riverpod` (v2.6.1) sont dans pubspec.yaml. L'app utilise `MaterialApp` avec `onGenerateRoute` (navigation imperative) et `StatefulWidget` avec `setState` (gestion d'etat manuelle).
**Probleme** : Les dependances non utilisees alourdissent le binaire et signalent une architecture inachevee. La navigation imperative ne permet pas de deep links ni de gestion propre du back stack. Le state management via setState dans un formulaire de 800 lignes devient fragile.
**Proposition** : Soit migrer vers go_router + Riverpod (si l'app va grossir), soit retirer les dependances inutilisees du pubspec.yaml pour ne pas creer de dette.

### 3. Melange tutoiement / vouvoiement
**Domaine** : Copy
**Choix actuel** : Home screen en tutoiement ("Verifie tes droits CAF"), formulaire et resultats en vouvoiement ("Votre situation familiale", "Vos droits estimes"), disclaimers en vouvoiement formel.
**Probleme** : L'incoherence de ton est perceptible des la premiere navigation. Pour une app traitant de sujets administratifs/juridiques, le vouvoiement inspire plus confiance. Le tutoiement sur la home peut paraitre deplace pour la cible (allocataires, pas que des jeunes).
**Proposition** : Passer au vouvoiement partout. Modifier le tagline en "Verifiez vos droits CAF en 3 minutes". Le ton peut rester accessible sans tutoyer.

### 4. Navigation vers /letter inexistante
**Domaine** : UX
**Choix actuel** : Le bouton "Contester — Generer un courrier" dans `results_screen.dart` navigue vers `/letter` (ligne 147). Cette route n'existe pas dans `onGenerateRoute` — elle tombe sur le `default` qui renvoie au HomeScreen.
**Probleme** : L'utilisateur qui detecte un ecart et clique sur le CTA principal revient a l'accueil sans feedback. C'est la pire rupture possible dans le parcours de conversion.
**Proposition** : Soit implementer l'ecran lettre, soit desactiver le bouton avec un message "Bientot disponible" et un CTA alternatif (email de notification quand la feature sera prete).

### 5. Pas de validation des champs avant changement d'etape
**Domaine** : UX
**Choix actuel** : Le bouton "Continuer" dans chaque etape du formulaire appelle `_nextStep()` sans aucune validation. On peut passer a l'etape 2 sans avoir saisi le nombre d'enfants, ou a l'etape 3 sans montant de revenu.
**Probleme** : L'utilisateur arrive a l'etape de calcul avec des donnees manquantes ou a zero, obtient un resultat faux (RSA max pour quelqu'un qui aurait oublie de saisir son revenu), et perd confiance dans l'outil.
**Proposition** : Ajouter une validation par etape : etape 1 (pas de validation requise, les defaults sont valides), etape 2 (si source revenu != aucun, montant > 0 requis), etape 3 (si locataire, loyer > 0 requis). Afficher un message inline, pas un snackbar.

### 6. Disclaimer trop petit et trop discret
**Domaine** : Copy / UX
**Choix actuel** : Le disclaimer legal sur le home screen est a 10px (`fontSize: 10`). Le disclaimer sur l'ecran resultats est a 11px. Pour un outil qui calcule des droits sociaux et genere des courriers juridiques, la visibilite du disclaimer est une obligation legale et une protection.
**Probleme** : Un disclaimer illisible ne protege pas juridiquement. Si un utilisateur envoie un courrier base sur un calcul errone, l'absence de disclaimer visible est un risque.
**Proposition** : Passer les disclaimers a 13px minimum. Sur le home, deplacer le disclaimer au-dessus du CTA (pas en dessous). Sur l'ecran resultats, encadrer le disclaimer dans une card distincte avec icone d'avertissement.

### 7. Baremes web Next.js incoherents avec la promesse produit
**Domaine** : Features
**Choix actuel** : Le site web (`alloccheck_web/`) est un scaffold Next.js vide — page.tsx affiche le template par defaut "To get started, edit the page.tsx file". Le layout.tsx a `title: "Create Next App"` et `lang="en"`.
**Probleme** : L'architecture prevoit des pages SEO (/simulateur-rsa, /simulateur-apl, /contester-caf) et un simulateur web. Rien n'est commence. Le metadata est celui du scaffold.
**Proposition** : Au minimum, mettre a jour le layout.tsx avec les bonnes metadata (`title: "AllocCheck — Verifiez vos droits CAF"`, `lang="fr"`), et creer une landing page statique meme sans simulateur. Les pages SEO sont la strategie d'acquisition principale — ne pas les repousser.

## Discutable — a ton appreciation

### 1. Rapport PDF et lettre au meme prix
**Domaine** : Monetisation
**Choix actuel** : Rapport PDF detaille = 4.99 EUR, lettre de contestation = 4.99 EUR.
**Probleme potentiel** : La lettre de contestation a plus de valeur percue (action concrete, references legales, genere par IA) que le rapport PDF (resume de ce que l'utilisateur a deja vu a l'ecran). Au meme prix, l'utilisateur choisit la lettre et ignore le rapport. Le rapport pourrait etre un produit d'appel a 1.99 EUR ou inclus gratuitement pour pousser vers la lettre.
**Alternative** : Rapport PDF = gratuit ou 1.99 EUR (lead magnet), lettre = 6.99 EUR (valeur percue plus haute), bundle rapport + lettre = 8.99 EUR.

### 2. Pas de preview du courrier avant paiement
**Domaine** : Monetisation / UX
**Choix actuel** : Le bouton "Contester — Generer un courrier" mene directement a la generation (qui sera payante). L'utilisateur ne sait pas a quoi ressemble un courrier avant de payer.
**Probleme potentiel** : La peur de l'inconnu freine la conversion. L'utilisateur ne sait pas s'il recevra un template generique ou un courrier personnalise.
**Alternative** : Montrer un apercu floute du courrier avec les 3 premieres lignes lisibles, le nom de l'aide et le montant reclame. Le paiement deverrouille le texte complet + le PDF.

### 3. Etape 4 ("Ce que je percois") obligatoire
**Domaine** : UX / Features
**Choix actuel** : L'etape 4 du formulaire demande systematiquement les montants percus. Le bouton "Calculer mes droits" est en bas de cette etape.
**Probleme potentiel** : Un utilisateur qui ne percoit rien actuellement (ou qui ne connait pas ses montants exacts) doit quand meme traverser cette etape. Le message "Si vous ne percevez aucune aide, laissez tout decoche" est present mais l'etape entiere pourrait sembler inutile.
**Alternative** : Ajouter un toggle en haut de l'etape 4 : "Je percois deja des aides CAF" / "Je ne percois rien actuellement". Si "rien", l'etape est skippee et l'app affiche directement les droits theoriques sans comparaison.

### 4. Pas de sauvegarde locale de la situation
**Domaine** : Features
**Choix actuel** : `shared_preferences` est en dependance mais non utilise. Si l'utilisateur quitte l'app avant le resultat, il perd toutes ses saisies.
**Probleme potentiel** : Le formulaire 4 etapes est long. Perdre ses donnees a l'etape 3 est frustrant, surtout avec les montants de revenus precis qu'il faut aller chercher.
**Alternative** : Sauvegarder la situation en cours dans SharedPreferences a chaque changement d'etape. Proposer "Reprendre ma simulation" au retour.

### 5. Zone logement sans aide a la determination
**Domaine** : UX / Copy
**Choix actuel** : 3 options radio — "Zone 1 — Paris et communes limitrophes", "Zone 2 — Agglomerations > 100 000 hab.", "Zone 3 — Reste de la France". Un info box dit "En cas de doute, selectionnez Zone 3".
**Probleme potentiel** : La zone APL est determinee par le code postal, pas par la taille de l'agglomeration. L'utilisateur peut se tromper (ex: Lyon est zone 1 depuis 2018, pas zone 2).
**Alternative** : Remplacer par un champ code postal et determiner la zone automatiquement (les tables de correspondance sont publiques sur legifrance.gouv.fr). Plus fiable et zero friction.

### 6. Edge Function AAH ne prend pas en compte la deconjugalisation
**Domaine** : Features
**Choix actuel** : Dans `calculate-rights/index.ts`, le calcul AAH utilise les ressources du foyer entier (`revenu_activite_demandeur + revenu_activite_conjoint + autres_revenus`). Le moteur local Dart, lui, ne prend que les ressources du demandeur (deconjugalisation correcte).
**Probleme potentiel** : Si l'Edge Function est utilisee (web ou fallback), les allocataires AAH en couple auront un calcul faux — potentiellement exclus a tort de l'AAH.
**Alternative** : Aligner l'Edge Function sur le calcul local : seules les ressources du demandeur comptent pour l'AAH depuis octobre 2023.

## Questions ouvertes

- Le CMG (Complement de libre choix du Mode de Garde) est mentionne dans CLAUDE.md comme aide couverte, mais absent du moteur de calcul. Est-ce prevu pour le MVP ou reporte ?
- Le schema DB prevoit un champ `situation JSONB` sans chiffrement reel (commentaire "en prod utiliser pgcrypto"). Quand le chiffrement AES-256 annonce dans les regles de securite sera-t-il implemente ?
- La strategie SEO repose sur des pages web non encore creees. Quel est le calendrier de mise en oeuvre du site Next.js ?
- L'app mobile n'a pas d'ecran de connexion/inscription. Les simulations sont-elles toutes anonymes en MVP, ou l'authentification est-elle prevue avant le lancement ?

## Top 3 actions immediates

1. **Unifier les baremes** : Supprimer ou desactiver l'Edge Function de calcul et utiliser exclusivement le moteur local Dart pour le MVP. Les baremes divergent sur toutes les aides, ce qui est inacceptable pour un outil de precision.
2. **Corriger le parcours contester** : Soit implementer l'ecran /letter, soit desactiver le bouton avec un etat "bientot disponible". Un CTA principal qui ramene au home est une rupture critique de conversion.
3. **Harmoniser le ton** : Passer au vouvoiement partout (home, formulaire, resultats, disclaimers). C'est une correction rapide qui ameliore immediatement la coherence percue.
