# CRITIQUE PRODUIT — AllocCheck v1.0.0
**Date** : 2026-04-06
**Cible** : Allocataires CAF (13,5M foyers) — personnes voulant vérifier/contester leurs droits
**Méthode** : Revue par 6 conseillers produit spécialisés (UX, Copy, Features, Onboarding, Monétisation, Cohérence)
**Résumé** : 18 observations (6 à corriger | 7 connus/acceptés | 5 points forts)

---

## Synthèse exécutive

L'app a significativement progressé depuis la dernière critique (04/02) : modèle de données restructuré avec 7 statuts conjugaux, pension alimentaire à 3 niveaux, 2 nouvelles aides (MVA, ASF), formulaire conditionnel intelligent, et génération PDF fonctionnelle. Le moteur de calcul local est solide et bien sourcé juridiquement.

Les axes de travail prioritaires sont : (1) l'absence de retour arrière natif dans le formulaire (le bouton physique Android ne navigue pas entre étapes), (2) le formulaire Step 1 trop long et non segmenté visuellement quand toutes les conditions sont actives (handicap + enfants + divorce + pension), (3) le paywall montre le chiffre d'écart mais pas le détail — le "early CTA" est bien placé mais le récap situation en section paywall ne convainc pas assez.

---

## Points forts — choix pertinents confirmés

### 1. Moteur de calcul 100% local
**Domaine** : Features
Le calcul se fait entièrement côté client sans appel réseau. Résultat instantané, pas de dépendance serveur, pas de coût par calcul. Les barèmes sont sourcés avec références légales précises (décrets, articles CSS/CASF). C'est un choix fort et pertinent.

### 2. Formulaire conditionnel intelligent
**Domaine** : UX
Le formulaire s'adapte dynamiquement au contexte : pension alimentaire n'apparaît que pour divorcé/séparé, mode de garde uniquement si enfant < 6 ans, MVA uniquement si AAH 80%+, etc. Réduit considérablement la complexité perçue.

### 3. Checklist "Autres revenus" avec montants fixes nationaux
**Domaine** : UX/Features
Les barèmes nationaux (ASS, bourses CROUS) sont pré-remplis automatiquement — l'utilisateur coche sans saisir. Réduit les erreurs et le temps.

### 4. Détection automatique zone APL par code postal
**Domaine** : UX
Le mapping code postal → zone APL (y compris codes postaux précis pour les agglos Zone 2) évite une question abstraite ("quelle est votre zone APL ?") que personne ne connaît.

### 5. Paywall bien positionné avec early CTA
**Domaine** : Monétisation
Le chiffre d'écart est visible gratuitement (hook émotionnel), le CTA apparaît immédiatement après, et le détail est derrière le paywall. Le prix à 2,99 EUR en paiement unique est correct pour la valeur perçue.

## À corriger — recommandations prioritaires

### 1. Bouton retour physique Android ne gère pas les étapes du formulaire
**Domaine** : UX
**Choix actuel** : Le `PageView` avec `NeverScrollableScrollPhysics` empêche le swipe, et le bouton retour de l'AppBar est bien implémenté (`_previousStep`). Mais le bouton physique/gesture retour Android (`Navigator.pop`) quitte la simulation au lieu de revenir à l'étape précédente.
**Problème** : Sur Android, un retour arrière involontaire fait perdre toute la saisie (pas de confirmation de sortie non plus).
**Proposition** : Wrapper le `SimulationScreen` dans un `PopScope` (anciennement `WillPopScope`) qui appelle `_previousStep` si `_currentStep > 0`, et affiche un dialogue de confirmation si `_currentStep == 0` et que des données ont été saisies.

### 2. Step 1 "Famille" surchargé quand toutes les conditions sont actives
**Domaine** : UX
**Choix actuel** : L'étape 1 contient potentiellement : statut conjugal (7 radios), compteur enfants, âge par enfant, pension versée/reçue/non perçue, mode de garde, congé parental, garde alternée, handicap, taux, situation de vie, besoin tierce personne.
**Problème** : Avec un profil divorcé + 3 enfants + handicap 80%, l'étape 1 fait 25+ widgets scrollables. L'utilisateur perd la notion de progression.
**Proposition** : Séparer en 2 étapes : "Famille" (statut + enfants + pension) et "Situation spécifique" (garde + handicap). Passer à 5 étapes au total. Alternative minimale : ajouter des séparateurs visuels (Divider + titre de section) entre chaque bloc conditionnel.

### 3. Validation pension versée manquante
**Domaine** : UX
**Choix actuel** : Si `_versePension` est coché mais le champ montant est vide ou à 0, le formulaire passe à l'étape suivante sans erreur (la validation Step 0 retourne toujours `null`).
**Problème** : L'utilisateur pense avoir déclaré sa pension mais le montant est 0, ce qui fausse le calcul RSA (la pension versée est déductible des ressources).
**Proposition** : Ajouter dans `_validateCurrentStep` case 0 : vérifier que si `_versePension` est true, le montant saisi est > 0.

### 4. Export PDF rapport ne fonctionne pas hors web
**Domaine** : Features
**Choix actuel** : Dans `_generateRapportPdf()` du `ResultsScreen`, le bloc `else` (non-web) contient `final _ = bytes;` — les bytes sont générés mais jamais partagés.
**Problème** : Sur iOS/Android, le bouton "Télécharger le rapport PDF" ne fait rien (pas d'erreur visible, pas de fichier). L'utilisateur paie 2,99 EUR et ne peut pas exporter.
**Proposition** : Ajouter `await Printing.sharePdf(bytes: bytes, filename: 'rapport_alloccheck.pdf');` dans le bloc non-web, comme c'est déjà fait dans `LetterScreen._exportPdf()`.

### 5. Token de paiement Stripe codé en dur et prévisible
**Domaine** : Monétisation
**Choix actuel** : `_validToken = 'AC2026UNLOCK'` — le token est un string statique dans le code source. N'importe qui peut appeler `PaymentService.unlockWithCode('AC2026UNLOCK')` ou ajouter `?paid=AC2026UNLOCK` à l'URL.
**Problème** : Le paywall est contournable en 5 secondes. Sur web, c'est visible dans le source JS compilé. Sur mobile, décompilable.
**Proposition** : Pour le MVP web, c'est acceptable tant que le volume est faible. Avant tout marketing sérieux : implémenter une vérification côté serveur via Supabase (Edge Function qui vérifie le Stripe payment_intent et retourne un token unique par session).

### 6. Garde alternée proposée uniquement aux couples
**Domaine** : Features
**Choix actuel** : `if (_isCouple)` conditionne l'affichage de la case "Garde alternée" (ligne 471 de simulation_screen.dart).
**Problème** : La garde alternée concerne principalement les parents séparés/divorcés, pas les couples. Un parent divorcé avec enfants ne voit jamais cette option.
**Proposition** : Inverser la condition : afficher la garde alternée pour les statuts `divorce`, `separe` (et éventuellement `celibataire` avec enfants), pas pour les couples.

## Connu/accepté — à l'appréciation de Joffrey

### 7. Pas d'onboarding ni d'explication du fonctionnement
**Domaine** : Onboarding
**Choix actuel** : L'écran d'accueil affiche le nom, le tagline "Vérifiez vos droits CAF en 3 minutes", 3 stats et un CTA.
**Problème** : L'utilisateur ne sait pas ce que l'app fait exactement ni comment elle le fait (calcul local vs officiel, quelles aides couvertes).
**Proposition** : Ajouter sous les stats une section "Comment ça marche" en 3 icônes : "1. Décrivez votre situation → 2. On calcule vos droits → 3. Contestez si écart". Pas un onboarding multi-écrans — juste une explication en 1 ligne.

### 8. Incohérence tutoiement/vouvoiement
**Domaine** : Copy
**Choix actuel** : L'app utilise le vouvoiement dans la majorité de l'interface ("Vos droits", "Votre situation", "Vos revenus"), mais le tagline dans `AppConstants` dit "Vérifie tes droits CAF" (tutoiement), et la description pubspec dit "Vérifie tes droits CAF et conteste si ça colle pas" (tutoiement familier).
**Problème** : Le vouvoiement est correct pour un outil institutionnel/juridique. Le tutoiement dans le tagline crée une dissonance.
**Proposition** : Unifier en vouvoiement partout. Tagline : "Vérifiez vos droits CAF". Description : "Vérifiez vos droits CAF et contestez en cas d'écart."

### 9. Labels Step 4 "Ce que je perçois" — mélange je/vous
**Domaine** : Copy
**Choix actuel** : Le titre d'étape dans la barre de progression dit "Ce que je perçois" (première personne), le titre de la page dit "Ce que la CAF vous verse" (deuxième personne vouvoiement).
**Problème** : Incohérence de point de vue dans le même écran.
**Proposition** : Unifier en "Ce que vous percevez actuellement" partout.

### 10. Disclaimer trop petit (fontSize 10-11)
**Domaine** : Copy/UX
**Choix actuel** : Le disclaimer juridique en bas de l'écran d'accueil est en fontSize 10 (`bodyMedium?.copyWith(fontSize: 10)`). Le disclaimer dans les résultats est en fontSize 11.
**Problème** : Sur un sujet juridique sensible (droits sociaux, contestation CAF), le disclaimer est la protection légale de l'app. Trop petit = pas lu = risque juridique.
**Proposition** : Minimum fontSize 12, dans un container visible (fond légèrement coloré comme déjà fait dans les résultats). Pas besoin d'être énorme, mais lisible.

### 11. Message d'erreur de calcul générique
**Domaine** : Copy
**Choix actuel** : L'écran d'erreur affiche `_error!` (le message technique de l'exception).
**Problème** : Le calcul local ne devrait jamais échouer (pas de réseau), mais si un bug survient, l'utilisateur voit un message technique incompréhensible.
**Proposition** : Wrapper dans un message humain : "Une erreur est survenue lors du calcul de vos droits. Réessayez ou contactez le support." avec un TextButton pour copier le détail technique.

### 12. Suggestions d'aides : CMG en doublon
**Domaine** : Features
**Choix actuel** : La CMG est calculée dans le moteur de calcul ET suggérée dans `_suggestionsAides` si enfant < 6 ans.
**Problème** : Un utilisateur qui a déclaré un mode de garde voit la CMG calculée dans ses droits ET dans les suggestions.
**Proposition** : Conditionner la suggestion CMG : ne l'afficher que si `droits.cmg == 0` (pas calculée parce que mode de garde = aucun mais enfant < 6 ans présent).

### 13. Courrier de contestation : "je saisie" au lieu de "je saisis"
**Domaine** : Copy
**Choix actuel** : Dans `_buildLetterText()`, le texte CRA contient "je saisie la Commission" (ligne 406 de letter_screen.dart).
**Problème** : Faute de conjugaison dans un courrier juridique formel.
**Proposition** : Corriger en "je saisis".
