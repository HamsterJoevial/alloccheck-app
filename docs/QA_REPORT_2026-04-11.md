## P0 — Bloquants (à corriger avant toute release)

### QA-001 · Faille sécurité — Token de validation hardcodé dans le bundle JS
**Type** : Faille sécurité
**Fichier** : `lib/core/services/payment_service.dart:17`
**Ce qui est vu** : `static const _validToken = 'AC2026UNLOCK';` — Cette constante est compilée dans le bundle web JavaScript. Tout utilisateur peut lire `main.dart.js` et extraire le token exact, puis forger l'URL `?paid=AC2026UNLOCK` pour débloquer sans payer.
**Impact** : 100% du paywall web contournable sans paiement, sans aucune compétence technique (1 recherche dans le bundle).
**Suggestion** : Stripe Webhook → Edge Function Supabase qui écrit un token unique chiffré en DB. La validation se fait côté serveur, pas dans le client. (Session dédiée — implique un backend.)

---

### QA-002 · Faille sécurité — Paywall entièrement client-side, bypassable via console
**Type** : Faille sécurité
**Fichier** : `lib/core/services/payment_service.dart`, `lib/core/utils/web_payment_bridge.dart`
**Ce qui est vu** : L'état débloqué (`ac_unlocked_sim`) est lu depuis `localStorage` côté client. N'importe qui peut ouvrir la console et exécuter `localStorage.setItem('flutter.ac_unlocked_sim', 'anysimid')` pour contourner le paywall.
**Impact** : Contournement sans paiement trivial pour tout utilisateur avec DevTools. Amplifie QA-001.
**Suggestion** : Même correction que QA-001 — validation serveur obligatoire.

---

### QA-003 · Bug — Enum CongeParental sérialisé par index (fragile aux refactorings)
**Type** : Bug
**Fichier** : `lib/features/simulation/screens/simulation_screen.dart` (sauvegarde draft)
**Ce qui est vu** : Le draft est sérialisé avec `.index` (entier) au lieu de `.name` (string). Si un nouveau cas est inséré **avant** un cas existant dans l'enum, tous les brouillons existants en localStorage seront désérialisés avec un mauvais état. L'ajout de `tauxPartiel` en fin d'enum (index 3) est safe aujourd'hui, mais le risque est latent.
**Impact** : Corruption silencieuse des données utilisateur après une mise à jour. Aucun crash immédiat — bug invisible jusqu'à découverte en prod.
**Suggestion** : Sérialiser par `.name` (string) dans `toJson/fromJson`. Ajouter un test de round-trip enum.

## P1 — Importants (à corriger rapidement)

### QA-004 · Bug — `fromJson()` sans `orElse` → crash sur données périmées
**Type** : Bug
**Fichier** : `lib/core/models/situation.dart` (et tous les `fromJson`)
**Ce qui est vu** : Les désérialisations d'enum utilisent `.firstWhere((e) => e.name == val)` sans clause `orElse`. Si la valeur stockée ne correspond plus à un cas de l'enum (après rename ou suppression), l'app crashe avec `StateError: No element`.
**Impact** : Crash au démarrage pour les utilisateurs qui ont un brouillon sauvegardé avec une ancienne version de l'app.
**Suggestion** : Ajouter `orElse: () => EnumType.defaultValue` sur tous les `firstWhere` d'enum.

---

### QA-005 · Bug — Génération PDF mobile est un stub non fonctionnel
**Type** : Bug
**Fichier** : `lib/core/services/pdf_service.dart` (ou équivalent)
**Ce qui est vu** : La génération PDF sur mobile contient `final _ = bytes;` — les bytes sont générés mais jamais partagés/sauvegardés. La feature est silencieusement cassée : l'utilisateur tape "Télécharger le rapport", rien ne se passe.
**Impact** : Feature payante non fonctionnelle sur iOS et Android. Fraude involontaire : l'utilisateur paie 0,99€ pour un rapport qu'il ne peut pas télécharger sur mobile.
**Suggestion** : Utiliser `share_plus` pour partager le PDF sur mobile, ou `path_provider` + ouverture avec le visualiseur système.

---

### QA-006 · Bug — `_clearDraft()` non awaité → race condition possible
**Type** : Bug
**Fichier** : `lib/features/simulation/screens/simulation_screen.dart`
**Ce qui est vu** : `_clearDraft()` est appelé sans `await` dans `_submitSimulation()`. Si la navigation vers ResultsScreen se produit avant la fin de l'effacement, un relancement de l'app pourrait restaurer un brouillon qui semblait avoir été soumis.
**Impact** : Brouillon fantôme potentiel — expérience confuse pour l'utilisateur.
**Suggestion** : `await _clearDraft()` avant la navigation.

---

### QA-007 · Bug — Race condition dans `ResultsScreen.initState`
**Type** : Bug
**Fichier** : `lib/features/results/screens/results_screen.dart`
**Ce qui est vu** : `_calculate()` et `_loadUnlockStatus()` sont lancés en parallèle dans `initState`. `_loadUnlockStatus` lit `isUnlockedForSim(simId)` et met à jour `_isUnlocked` avant que le calcul soit terminé — le widget peut se builder en état intermédiaire incohérent (calculs affichés = null, isUnlocked = true).
**Impact** : Flash d'affichage anormal, potentiellement un paywall qui s'affiche brièvement même pour un utilisateur ayant payé.
**Suggestion** : Séquencer : `await _calculate()` puis `_loadUnlockStatus()`, ou utiliser un `Future.wait` explicite.

---

### QA-008 · Bug — `jsonDecode` sans try/catch dans PaymentService
**Type** : Bug
**Fichier** : `lib/core/services/payment_service.dart:82, 106`
**Ce qui est vu** : `getSavedSituation()` et `getLastSimulation()` appellent `jsonDecode(jsonStr)` sans protection. Si le localStorage est corrompu (app kill pendant l'écriture, données manipulées manuellement), l'app crashe avec `FormatException`.
**Impact** : Crash au démarrage si localStorage corrompu — utilisateur bloqué.
**Suggestion** : Encapsuler dans try/catch, retourner `null` en cas d'erreur et supprimer la clé corrompue.

---

### QA-009 · Friction UX — Brouillon non affiché si `nombreEnfants == 0`
**Type** : Friction UX
**Fichier** : `lib/features/simulation/screens/simulation_screen.dart`
**Ce qui est vu** : La logique de restauration du brouillon conditionnée à `nombreEnfants > 0` (ou équivalent). Les célibataires sans enfant — cible principale de RSA, prime d'activité — ne voient pas leur brouillon proposé à la reprise.
**Impact** : Perte de données perçue pour le cas d'usage le plus fréquent. Friction majeure en reprise de session.
**Suggestion** : La restauration du brouillon ne doit pas dépendre du contenu — elle doit s'afficher si une clé de draft existe en localStorage, quelle que soit la situation.

---

### QA-010 · Dette technique — Zéro tests sur PaymentService
**Type** : Dette technique
**Fichier** : `test/` (absent)
**Ce qui est vu** : `PaymentService` est le composant le plus critique du projet (monétisation) et n'a aucun test. Les méthodes `isUnlockedForSim`, `checkUrlAndUnlock`, `saveSituationAndOpenStripe` ne sont pas couvertes.
**Impact** : Toute régression sur le paywall passera inaperçue jusqu'en prod.
**Suggestion** : Ajouter des tests unitaires avec mock de SharedPreferences. Couvrir : unlock via URL, persistence simId, isUnlockedForSim.

---

### QA-011 · Accessibilité — `localizationsDelegates` absent → TalkBack en anglais
**Type** : Accessibilité
**Fichier** : `lib/main.dart` (MaterialApp)
**Ce qui est vu** : `MaterialApp` ne déclare pas `localizationsDelegates: GlobalMaterialLocalizations.delegates`. Les widgets Material natifs (RadioListTile, Checkbox, Dialog) utilisent leurs labels anglais par défaut pour les screen readers : "checked", "unchecked", "Close dialog".
**Impact** : App inutilisable en accessibilité pour les utilisateurs francophones utilisant TalkBack/VoiceOver.
**Suggestion** : Ajouter `localizationsDelegates: GlobalMaterialLocalizations.delegates` + `supportedLocales: [Locale('fr', 'FR')]` dans MaterialApp.

## P2 — Améliorations (planifier)

### QA-012 · Bug — Accents manquants dans les PDF générés (9+ occurrences)
**Type** : Bug
**Fichier** : `lib/core/services/pdf_service.dart` (templates de courriers)
**Ce qui est vu** : Les courriers PDF contiennent des mots sans accents : "reclamee" (réclamée), "theorique" (théorique), "regularisation" (régularisation), "reference" (référence), etc. La police ou l'encodage ne supporte pas les caractères accentués.
**Impact** : Courriers de contestation avec fautes d'orthographe — perte de crédibilité vis-à-vis de la CAF.
**Suggestion** : Utiliser une police TrueType avec support Unicode complet (ex: Roboto depuis les assets Flutter), ou substituer les caractères accentués avant génération.

### QA-014 · Bug — ARS affichée en équivalent mensuel (montant trompeur)
**Type** : Bug
**Fichier** : `lib/core/services/calcul_local_service.dart` (affichage ARS)
**Ce qui est vu** : L'Allocation de Rentrée Scolaire est une aide **annuelle versée en août**. Le code l'affiche en équivalent mensuel divisé par 12, donnant l'impression d'un montant mensuel continu. Ex: "ARS : 42€/mois" au lieu de "ARS : 508€/an (versement unique en août)".
**Impact** : Utilisateur qui calcule son budget mensuel sur une aide qu'il ne recevra qu'une fois par an. Erreur financière réelle.
**Suggestion** : Afficher le montant annuel brut avec la mention "versement unique en août".

### QA-021 · Robustesse — Pas de gestion du cas `webSaveSituationAndNavigate` qui échoue
**Type** : Robustesse
**Fichier** : `lib/core/utils/web_payment_bridge.dart`, `lib/core/services/payment_service.dart:64-69`
**Ce qui est vu** : L'appel JS `webSaveSituationAndNavigate()` ne peut pas remonter d'erreur à Dart (interop fire-and-forget). Si localStorage est plein ou bloqué (navigation privée Firefox), la situation n'est pas sauvegardée mais Stripe s'ouvre quand même. Au retour, `getSavedSituation()` retourne null.
**Impact** : L'utilisateur paie mais son rapport n'est plus disponible au retour.
**Suggestion** : Vérifier la capacité localStorage avant navigation, afficher un warning si indisponible, et proposer une alternative (mémoriser le simId dans l'URL Stripe via `client_reference_id`).

### QA-022 · Sécurité — `_pendingSimKey` jamais nettoyé si paiement abandonné
**Type** : Robustesse / Sécurité
**Fichier** : `lib/core/services/payment_service.dart`
**Ce qui est vu** : Si l'utilisateur ouvre Stripe puis abandonne le paiement et revient sur l'app sans payer, `_pendingSimKey` reste en localStorage. Si une autre simulation est lancée et que l'utilisateur revient sur l'URL Stripe d'une session précédente avec un token valide, le simId peut être incorrect.
**Impact** : Déverrouillage d'une mauvaise simulation (cas improbable mais possible).
**Suggestion** : Nettoyer `_pendingSimKey` au démarrage de l'app si aucun token n'est détecté dans l'URL.

## Checklist de test manuel sur device

> Ces scénarios ne peuvent être validés que sur un device ou émulateur réel.

### Flow principal
- [ ] Remplir une simulation complète (célibataire, 0 enfant) → vérifier que les droits s'affichent correctement
- [ ] Taper "Obtenir mon rapport" → vérifier la redirection Stripe sur le bon lien
- [ ] Effectuer un vrai paiement de test (carte Stripe test) → vérifier que le rapport se débloque au retour
- [ ] Quitter l'app après paiement → relancer → vérifier que la simulation reste débloquée (même simId)
- [ ] Lancer une **nouvelle** simulation après paiement → vérifier que le paywall réapparaît (nouveau simId)

### Brouillon
- [ ] Remplir la simulation à moitié → fermer l'app → relancer → vérifier que "Reprendre" s'affiche
- [ ] Cas célibataire sans enfant : vérifier que le brouillon est bien proposé (QA-009)

### PDF / Courrier
- [ ] Sur mobile (iOS) : tenter de télécharger le rapport → vérifier qu'un fichier s'ouvre réellement (QA-005)
- [ ] Ouvrir le PDF → vérifier les accents dans le courrier (QA-012)

### Accessibilité
- [ ] Activer TalkBack (Android) → naviguer dans la simulation → vérifier que les RadioListTile annoncent en français (QA-011)
- [ ] Mode texte grand (200%) → vérifier que les montants et labels ne se coupent pas

### Sécurité
- [ ] (Web) Ouvrir DevTools → localStorage → chercher `ac_unlocked_sim` → modifier manuellement → vérifier que l'app ne débloque pas un rapport sans paiement (QA-001, QA-002)
